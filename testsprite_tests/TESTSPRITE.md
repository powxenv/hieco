# TestSprite Setup

## Scope

- Primary app under test: `apps/web`
- Public package behaviors exposed through `apps/web/src/routes/testsprite-lab.tsx`
- Test root: repository root

## Local Run

```bash
bun --filter web build
bun --filter web preview:testsprite
```

The preview server listens on `http://localhost:5878`.

## Bootstrap Inputs

- `projectPath`: repository root
- `type`: `frontend`
- `testScope`: `codebase`
- `localPort`: `5878`
- `pathname`: `/testsprite-lab`

## High-Value Coverage

- `/`
  - Header navigation
  - Connect wallet dialog structure
  - Demo tab switching across core package surfaces
- `/showcase`
  - Search input and URL sync
  - Package, use-case, and open-source filters
  - Deep-linked query hydration
  - Unauthenticated submission guard
- `/showcase/not-a-real-project`
  - Missing project fallback
- `/testsprite-lab`
  - `@hieco/wallet` state helpers
  - `@hieco/sdk` error classification and unwrap behavior
  - `@hieco/utils` entity and network validation
  - `@hieco/mirror` base URL configuration
  - `@hieco/realtime` pre-connect subscription guard

## Do Not Force Through TestSprite

- Real wallet extension installation and signing
- Convex mutations that require live credentials and signed ownership challenges
- External Mirror Node availability or response-shape drift
- Live websocket relay behavior against a real server
- Thin wrapper packages with no standalone runtime surface:
  - `packages/mirror-react`
  - `packages/mirror-preact`
  - `packages/mirror-solid`
  - `packages/react`
  - `packages/realtime-react`
  - `packages/mirror-cli`
  - `packages/mirror-mcp`

## Execution Notes

- Use the generated TestSprite plan to prioritize meaningful failure paths over happy-path duplication.
- Keep generated coverage focused on route-level behavior and public package contracts visible through the harness.
- If tests fail because of product defects, fix the implementation and rerun the same tests instead of weakening them.
- Previous mismatch findings:
  - The old `TC004` failure was tied to an earlier timing-sensitive search implementation plus an immediate URL assertion in generated test code.
  - The old `TC010` failure was not trustworthy evidence of a product bug: one generated version never changed the wallet scenario, and a later regenerated version passed while asserting only that the page URL existed.
  - Because of that, the current curated plan uses exact visible control labels and deterministic assertions instead of trusting stale autogenerated steps.
